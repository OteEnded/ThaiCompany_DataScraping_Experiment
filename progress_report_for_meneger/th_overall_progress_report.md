
# รายงานความคืบหน้าโครงการวิจัย (ภาพรวม)

วันที่รายงาน: 2026-04-03

ชื่อโครงการ: ThaiCompany_DataScraping_Experiment

## 1) บทสรุปสำหรับผู้บริหาร (พี่บอย)

โครงการมีความคืบหน้าอย่างต่อเนื่อง และปัจจุบันสามารถเก็บข้อมูลจากหลายแหล่งได้ครบตามกรอบการทดลอง ได้แก่ DBD DataWarehouse, Settrade และ AI Search orchestration โดยมีจุดเด่นสำคัญอยู่ที่ Process F ซึ่งพัฒนาจากการ scrape ผ่านหน้าเว็บอย่างเดียว ไปสู่โครงสร้างแบบ hybrid (UI + API replay) ที่เสถียรมากขึ้นอย่างชัดเจน

หมายเหตุเชิงประเภทงาน:

- Process F จัดเป็นกระบวนการลักษณะ RPA (Robotic Process Automation) สำหรับงานเก็บข้อมูลเชิงเว็บและระบบหลังบ้าน โดยมี automation flow, เงื่อนไขตัดสินใจ, retry/recovery, และการบันทึกหลักฐานการทำงาน
- - สำหรับดึงข้อมูล List ของบริษัทจาก DBD

  

ผลสำคัญล่าสุด:

- Process F ผ่านการทดสอบแบบมีเงื่อนไขกรองหลายหน้าได้สำเร็จ (ก่อนหน้านี้ยังทำระบบกรองไม่สำเร็จ เพราะหน้าเว็ปบังคับให้คลิกเข้าไปเลือก Menu Filter เอง เลยต้องเขียน RPA ส่วนนี้เพิ่ม)

- รอบทดสอบล่าสุดแบบ 18 หน้า ได้ข้อมูล 180 บริษัท และสถานะผลลัพธ์เป็น ok และพร้อมจะปล่อยให้ Program run ไปเรื่อยๆ เพื่อเก็บข้อมูลได้

- มีระบบ **รับมือความไม่เสถียรของเว็บและ API** (timeout/500/502) พร้อม retry และ fallback

  

สถานะโดยรวม:

- อยู่ในช่วง "ใช้งานจริงเชิงงานวิจัยได้" และกำลังเก็บงาน hardening ขั้นสุดท้าย

  

## 2) วัตถุประสงค์โครงการ

1. สร้างระบบทดลองเก็บและสรุปข้อมูลบริษัทไทยจากหลายแหล่งในโครงสร้างเดียวกัน (ผมลองจากทั้ง DBD และ Settrade)

2. ทดสอบความเป็นไปได้ของ pipeline ตั้งแต่ค้นหา, ดึงข้อมูลดิบ, ถอดรหัส/จัดรูป, สรุปผล, และส่งออก

3. พยายามแก้ปัญหาของการดึงข้อมูลจากเว็ปที่มี anti-bot และหน้าเว็บเปลี่ยนพฤติกรรมได้

## 3) ภาพรวมความคืบหน้าตามกระบวนการ

### Process A: AI Search

- ทดลองใช้ AI หาข้อมูลบริษัทจาก Internet เหมือนกับพิมพ์ถามใน ChatGPT

- พัฒนา flow ค้นหาด้วย Brave + SiliconFlow สำเร็จ

- มีผลลัพธ์และ dump สำหรับตรวจสอบย้อนหลัง

- (ยังไม่ค่อย Work เลยพับไว้ก่อนครับ)  

### Process B: DBD Scraper (รายบริษัทตามเลขนิติบุคคล)

- ดึงข้อมูลบริษัทและงบการเงินตามเลขจดทะเบียน

- พัฒนา flow เข้าหน้าบริษัทและถอดรหัส payload สำเร็จ

- มีการจัดการสถานะ blocked/partial จาก anti-bot

- ส่งออกผล JSON ดิบ + ถอดรหัส (ตัว DBD Encrypt ข้อมูลก่อนขึ้นหน้าเว็ปไว้ ผมเลยต้องไปแกะวิธีถอดข้อมูลมา)


### Process C: DBD AI Summary

- AI Pipeline/Flow สรุปข้อมูลจาก Process B

- เชื่อมข้อมูลจาก B เพื่อสรุปเชิงวิเคราะห์ได้

- ส่งออก compact JSON + markdown summary

### Process D: Settrade SDK

- เชื่อม SDK สำเร็จสำหรับข้อมูลตลาด/บัญชี (ขึ้นกับ credential)

- (ข้อมูลที่ได้ ไม่เกี่ยวข้องกับงาน ตัว SDK มีแค่ข้อมูล Port หุ้น)

### Process E: Settrade Scraper

- ดึงข้อมูลสาธารณะราย symbol ได้ (profile/info/overview/shareholder/historical/corporate action)

- (ได้ข้อมูลไม่ครบ มีเฉพาะจ้อมูลตลาดหุ้น)

### Process F: DBD Company List With Filter (โฟกัสหลัก)

- พัฒนาจนเป็นแกนหลักของงานวิจัยด้าน list extraction แบบมีเงื่อนไข

- รองรับการดึงข้อมูลแบบหลายหน้าและการกรองขั้นสูง

- มี telemetry/logging สำหรับวิเคราะห์ run และปัญหา

## 4) รายละเอียดความคืบหน้า Process F

Process F เป็น RPA workflow สำหรับงาน data extraction โดยผสานการควบคุมหน้าเว็บ (UI automation) กับการเรียก API เชิงโปรแกรม (API automation) ภายใต้กติกาการควบคุมความเสถียรของรันไทม์ (เว็ปบั๊ค/ เว็ปโหลดช้า)


## 4.1 สิ่งที่ทำเสร็จแล้ว

1. ออกแบบสถาปัตยกรรม hybrid

- ใช้ UI สำหรับเปิดผลค้นหา/ตั้งค่า filter

- ใช้ API replay สำหรับดึงหน้าถัดไปอย่างมีประสิทธิภาพ

  

2. เพิ่มความเสถียรการทำงาน

- direct URL-first search ก่อน fallback กลับไป search box

- filter apply แบบ retry + refresh recovery

- overlay-aware interaction ลดโอกาสคลิกไม่ติด

- timeout/retry/backoff สำหรับ replay API

  

3. เพิ่มการคุมคุณภาพข้อมูล

- guard ตรวจว่า replay body มี filter keys ครบ

- ถ้า body ที่จับได้ไม่ครบ จะ rebuild payload จาก config mapping

- deduplicate ด้วย juristic_id และ fallback profile key

  

4. แก้ปัญหา pagination/sort ที่พบจริง

- แก้ bug pages = -1 (fetch-all)

- เพิ่ม fetch_all_max_pages เป็น safety cap

- เพิ่ม total pages hint + last page guard

- แก้กรณีจังหวัด (pvDesc) ที่ pagination ไม่เสถียรด้วย stable workaround

  

5. เพิ่มการสังเกตการณ์และการตรวจสอบย้อนหลัง

- last_run.log แบบมี timestamp

- last_page_on.png สำหรับสถานะค้างหน้า UI

- timing summary ต่อหน้า/ทั้งรัน

- CSV แบบ streaming ระหว่างรัน ลดความเสี่ยงข้อมูลหายเมื่อรันยาว

  

## 4.2 ผลการทดสอบล่าสุด (สำคัญ)

- ทดสอบ filtered run 18 หน้า: สำเร็จ

- จำนวนข้อมูลสุดท้าย: 180 บริษัท

- สถานะผลลัพธ์: ok

- พบ error ชั่วคราวจากปลายทาง (500/502/timeout) แต่ระบบ retry/fallback กู้รันได้

  

## 5) กระบวนการทำงานปัจจุบัน (Process F)

1. เปิดเว็บ DBD และโหลดผลค้นหา (direct URL ก่อน)

2. เปิดแผง filter และใส่เงื่อนไข

3. จับ request contract ของ endpoint infos

4. ตรวจสอบ/ซ่อม replay payload ให้สอดคล้อง filter

5. replay หน้า 2..N พร้อม retry และ pacing (ดึงข้อมูลจาก หน้าเว็ป)

6. fallback ด้วย UI probe เมื่อ API ช้า/ล้มเหลว (เว็ปบั๊ค/ เว็ปโหลดช้า)

7. deduplicate, post-process, จัดเรียงผลตามเงื่อนไข

8. ส่งออก JSON/CSV พร้อม log และ debug artifacts

## 6) ความเสี่ยงและข้อจำกัดที่ยังเหลือ

ความเสี่ยงหลัก:

- แผง filter บางช่วงยังโหลดช้า/ไม่พร้อมคลิก ทำให้เสียเวลาในรอบ recovery

  
ความเสี่ยงรอง:

- API ปลายทางยังมีความไม่เสถียรเป็นช่วงๆ

- sort แบบจังหวัดฝั่ง API เดิมยังมีปัญหา pagination จึงต้องใช้ workaround ต่อ

  
ผลกระทบ:

- อาจมีบางรอบที่ใช้เวลานานกว่าปกติ หรือจบเป็น partial หาก UI ไม่พร้อมติดต่อกันหลายรอบ

  

## 7) แผนงานถัดไป (ลำดับความสำคัญ)

1. เพิ่ม auto-selection สำหรับ sort ที่เสถียร เมื่อผู้ใช้ต้องการจัดเรียงตามจังหวัด

2. เพิ่ม post-run assertions ตรวจว่าแถวผลลัพธ์ยังตรงกับเงื่อนไข filter ที่ตั้งไว้

3. hardening เพิ่มในช่วงเปิด/ใช้งาน filter panel ใต้สภาพเว็บช้า

4. ทดสอบ bounded fetch-all เพิ่มเติมหลายชุด เพื่อวัดคุณภาพและเวลา

5. เพิ่ม checkpoint/resume สำหรับงานรันยาว

  

## 8) รายงานสถานะสำหรับผู้บริหาร (พี่บอย)

สถานะรวม: เดินหน้าได้ตามแผน และความเสี่ยงถูกควบคุมในระดับใช้งานวิจัย


สิ่งที่สำเร็จแล้ว:

- โครงสร้าง pipeline หลายแหล่งข้อมูลพร้อมใช้งาน

- Process F มีความเสถียรสูงขึ้นอย่างมีนัยสำคัญ

- มีผลทดสอบยืนยันแบบหลายหน้าและมีระบบป้องกันข้อมูลผิดเงื่อนไข


สิ่งที่กำลังเก็บงาน:

- ความเสถียรเชิง UI บางช่วงที่โหลดช้า

- เสริม validation/checkpoint ให้รองรับงานรันยาวมากขึ้น


ระดับความมั่นใจ:

- สูง สำหรับรันแบบควบคุมเงื่อนไขและจำนวนหน้า

- ปานกลางถึงสูง สำหรับรันยาวภายใต้สภาพเว็บไม่เสถียร


## 9) ข้อเสนอรูปแบบรายงานรอบถัดไป

- รายงานสั้นรายวัน: success rate, เวลาเฉลี่ย, จำนวน retry, คุณภาพข้อมูล

- รายงานสรุปรายสัปดาห์: milestone, แนวโน้มความเสี่ยง, แผนสัปดาห์ถัดไป


## 10) สรุป 1 บรรทัด

โครงการมีความคืบหน้าชัดเจนและ Process F อยู่ในระดับพร้อมใช้งานจริงเชิงงานวิจัย โดยเหลือการ hardening ขั้นสุดท้ายด้านความเสถียร UI และความทนทานของงานรันยาว.


นิยามสั้นสำหรับผู้บริหาร:

- Process F คือ RPA ด้าน Data Collection ที่ออกแบบให้ทำงานซ้ำได้, ตรวจสอบย้อนหลังได้, และฟื้นตัวจากความผิดพลาดระหว่างรันได้อัตโนมัติ
